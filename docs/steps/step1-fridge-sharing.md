# 冷蔵庫共有・招待機能 要件定義

> Step 1 DB スキーマ設計に先立ち、冷蔵庫の共有モデルを定義する。

---

## 1. 概要

複数のユーザーが 1 つの冷蔵庫を共有し、食材の追加・消費・閲覧を行える仕組み。
家族やルームメイトなど、同じ冷蔵庫を使う人が同一データにアクセスできるようにする。

---

## 2. 基本ルール

| 項目 | 仕様 |
|---|---|
| 冷蔵庫の所有 | 1 ユーザーにつき 1 冷蔵庫のみ所属可能 |
| 権限 | 全メンバー同等（オーナー/メンバーの区別なし） |
| 招待方法 | 招待リンク（URL）を共有 |
| 食材の所属 | 食材はユーザーではなく冷蔵庫に紐づく |

---

## 3. ユーザーフロー

### 3-1. 新規ユーザー（冷蔵庫を持っていない場合）

```
初回ログイン
  → 冷蔵庫を自動作成（名前: デフォルト「マイ冷蔵庫」）
  → ユーザーをメンバーとして登録
  → ダッシュボードへ遷移
```

- 初回ログイン時に冷蔵庫が自動的に 1 つ作成される
- ユーザーが意識しなくても「冷蔵庫グループ」が裏側で作られる

### 3-2. 招待リンクの発行

```
設定画面 → 「メンバーを招待」ボタン
  → 招待リンクを生成
  → リンクをコピー（LINE / メール等で共有）
```

- 招待リンクは冷蔵庫ごとに 1 つ
- リンクにはトークン（UUID）を含む（例: `/invite/{token}`）
- リンクの有効期限: 7 日間（期限切れ後は再発行が必要）
- 同時に有効なリンクは 1 つまで（再発行すると旧リンクは無効化）

### 3-3. 招待を受ける（参加フロー）

```
招待リンクをタップ
  → ログイン済み？
     → Yes → 確認画面「○○さんの冷蔵庫に参加しますか？」
              → 参加ボタン → 冷蔵庫切り替え → ダッシュボードへ
     → No  → ログイン画面 → ログイン後に確認画面へ
```

#### 参加時の処理

- 参加者が既に別の冷蔵庫に所属している場合:
  - 既存の冷蔵庫から離脱する
  - 既存の冷蔵庫に他のメンバーがいなければ、その冷蔵庫と食材を削除する
  - 既存の冷蔵庫に他のメンバーがいれば、冷蔵庫はそのまま残る
- 参加者は新しい冷蔵庫のメンバーになる
- 1 ユーザー 1 冷蔵庫のルールを常に維持する

### 3-4. メンバーの離脱

```
設定画面 → 「冷蔵庫から離脱」ボタン
  → 確認ダイアログ「本当に離脱しますか？」
  → 離脱 → 新しい冷蔵庫を自動作成 → ダッシュボードへ
```

- 離脱後は空の冷蔵庫が自動作成される
- 離脱した冷蔵庫に他のメンバーがいなければ、冷蔵庫と食材を削除する

---

## 4. 招待リンクの仕様

| 項目 | 仕様 |
|---|---|
| URL 形式 | `https://{domain}/invite/{token}` |
| トークン形式 | UUID v4 |
| 有効期限 | 7 日間 |
| 同時有効数 | 冷蔵庫 1 つにつき 1 リンク |
| 再発行 | 旧リンクを無効化し新トークンを発行 |
| 使用回数制限 | なし（リンクを知っている人は誰でも参加可能） |

---

## 5. データモデルへの影響

### 現行（Step 1 の元設計）

```
User ──< Food    （ユーザーが食材を所有）
```

### 変更後

```
User ──< FridgeMember >── Fridge ──< Food
                          Fridge ──< FridgeInvitation
```

- **Food の所属先が User → Fridge に変わる**
- User は FridgeMember を経由して Fridge に所属する
- FridgeInvitation で招待リンクのトークン・有効期限を管理する

### 新規テーブル

| テーブル | 用途 |
|---|---|
| `Fridge` | 冷蔵庫グループ |
| `FridgeMember` | 冷蔵庫とユーザーの紐づけ |
| `FridgeInvitation` | 招待リンクのトークン管理 |

### 既存テーブルへの変更

| テーブル | 変更内容 |
|---|---|
| `Food` | `userId` → `fridgeId` に変更 |
| `CategoryStorageMapping` | `userId` → `fridgeId` に変更（保管場所設定は冷蔵庫単位） |

---

## 6. 同時操作の考慮

- 複数ユーザーが同時に食材を追加・消費する可能性がある
- PostgreSQL は行レベルロックを持つため、同一レコードの同時更新は DB レベルで直列化される
- 消費操作（数量の減算）は `updatedAt` による楽観的ロックを適用する
  - 更新時に `WHERE id = ? AND updatedAt = ?` で競合を検知
  - 競合時はエラーを返し、再取得を促す
- 追加操作は競合しない（新規レコードの INSERT）

---

## 7. Phase 分け

| Phase | 実装範囲 |
|---|---|
| Phase 1 | DB に Fridge / FridgeMember / FridgeInvitation テーブルを作成。初回ログイン時に冷蔵庫を自動作成。Food は fridgeId で管理。招待機能は Phase 1 後半または Phase 2 で実装。 |
| Phase 1 後半 | 招待リンク発行・参加・離脱の UI と API を実装 |

### Phase 1 での最小実装

- Fridge テーブル作成
- FridgeMember テーブル作成
- FridgeInvitation テーブル作成（構造だけ、API は後半で）
- ログイン時に Fridge 自動作成 + FridgeMember 自動登録
- Food.fridgeId で食材を冷蔵庫に紐づけ
- API は fridgeId を使って食材を取得・追加

---

## 8. API エンドポイント（概要）

| メソッド | パス | 用途 | Phase |
|---|---|---|---|
| POST | `/api/fridge/invite` | 招待リンク発行 | 1 後半 |
| GET | `/api/fridge/invite/{token}` | 招待情報の取得（確認画面用） | 1 後半 |
| POST | `/api/fridge/join/{token}` | 冷蔵庫に参加 | 1 後半 |
| POST | `/api/fridge/leave` | 冷蔵庫から離脱 | 1 後半 |
| GET | `/api/fridge/members` | メンバー一覧取得 | 1 後半 |

---

## 9. 画面（概要）

| 画面 | 内容 | Phase |
|---|---|---|
| 設定画面（拡張） | メンバー一覧、招待リンク発行ボタン、離脱ボタン | 1 後半 |
| 招待確認画面 | `/invite/{token}` — 冷蔵庫名と参加ボタンを表示 | 1 後半 |
