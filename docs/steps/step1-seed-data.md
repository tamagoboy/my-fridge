# Step 1: シードデータ設計

> 開発・動作確認・テスト用のシードデータ定義

---

## 概要

Step 3（食材一覧画面）以降の UI 開発・テストをスムーズに進めるため、全テーブルにシードデータを投入する。

---

## 作成ファイル

| ファイル | 用途 |
|---|---|
| `prisma/seed.ts` | シードデータ投入スクリプト |

---

## 設計方針

- **冪等性**: `upsert` を使い、何度実行しても同じ結果になるようにする
- **固定 ID**: テストで参照しやすいよう、主要レコードには固定 ID を使用
- **現実的なデータ**: 実際の食材名・カテゴリ・保管場所を使い、UI の見た目を確認しやすくする
- **期限切迫度の網羅**: 赤（期限切れ・当日）、黄（3 日以内）、緑（4 日以上）、null（期限なし）を含める

## 7. 実行手順

```bash
# シードデータ投入
npm run db:seed

# 確認
npm run db:studio
```

---

## 8. 確認項目

| 確認内容 | 方法 |
|---|---|
| `npm run db:seed` がエラーなく完了する | コマンド実行 |
| 再実行しても重複エラーにならない（冪等性） | 2 回連続で実行 |
| Prisma Studio で全テーブルのデータが確認できる | `npm run db:studio` |
| Food が保管場所別に正しく分類されている | Prisma Studio でフィルタ |
| 期限切迫度（赤/黄/緑/なし）が網羅されている | Prisma Studio で expiryDate 確認 |

---

## 9. 注意事項

- `googleId` はダミー値。本番環境では Google OAuth の `sub` フィールドが入る
- シードは開発・テスト環境専用。本番環境では `npm run db:seed` を実行しないこと
- 賞味期限は `seed.ts` 実行時の日付を基準に相対的に計算する（固定日付にしない）
