# 冷蔵庫管理アプリ 要件定義書 (Draft 3.0)

## 1. プロジェクト概要

* **目的**: 冷蔵庫内の食材を管理し、賞味期限切れによる廃棄を防ぐとともに、買い物の効率化を図る。
* **ターゲット**: 自炊を行う単身者およびファミリー層。
* **コア価値**: 「入力の手間を極限まで減らす（スマート入力）」こと。

## 2. システム構成・対応プラットフォーム

* **提供形態**: Webアプリ（PWA対応）
  * **採用技術**: Next.js (React / TypeScript)
  * **変更理由**: iOSでは個人開発者がアプリを無料で配布し続けることが困難（Apple Developer Program年額$99 + 審査コスト）。Webアプリ（PWA）にすることで、ストア審査不要・無料で継続運用が可能。
  * **PWAによるネイティブ相当の体験**: ホーム画面追加、オフライン対応、プッシュ通知（iOS Safari 16.4+ / Android Chrome 対応）
* **対応ブラウザ**:
  * **iOS**: Safari（PWA機能をフル活用するため。iOSでは全ブラウザがWebKitエンジンのため、Safari が最も機能制限が少ない）
  * **Android**: Google Chrome
  * **PC**: Google Chrome（今後 Safari / Microsoft Edge にも対応予定）
  * **方針**: クロスブラウザ互換性を意識した実装を行い、段階的にブラウザ対応を拡大
* **レスポンシブ対応**: モバイルファースト設計

## 3. 機能要件 (フェーズ別)

### Phase 1: コア機能（プロトタイプ / MVP）

このフェーズではレシートOCRを中心とした「実用的な最小機能」を完成させます。プロトタイプとして動作する状態を作り、そこから段階的に機能を追加していきます。

| 機能ID | 機能名 | 詳細要件 | 優先度 |
|---|---|---|---|
| F-01 | レシートOCR入力 | ・カメラまたはギャラリーからレシート画像を取得<br>・OCR処理で商品名、数量、金額を自動認識<br>・**OCRエンジン**: Google Cloud Vision API を使用（サーバーサイド処理）<br>・**カメラ**: `<input type="file" capture="environment">` でOS標準カメラを起動<br>・認識後、即座に商品名正規化処理を実行<br>・確認画面で編集可能<br>・DBへの一括保存 | **最優先** |
| F-02 | 商品名正規化処理 | **OCR読み取り直後に以下の処理をサーバーサイドで自動実行**:<br>1. ひらがな・カタカナを全てカタカナに統一<br>2. 店舗固有の修飾語を除去（例: 「おいしい豚肉」→「豚肉」）<br>3. 英語表記を日本語に変換（例: 「ポーク」→「豚」、「ビーフ」→「牛」）<br>・変換辞書はPostgreSQLに保持<br>・正規化後の商品名でDB検索し、既存マスタとマッチング | **最優先** |
| F-03 | 食材一覧表示 | ・保管場所別タブ（冷蔵室、冷凍室、野菜室、常温）<br>・賞味期限昇順での表示<br>・期限切迫度に応じたカラー表示（赤/黄/緑）<br>・基本的な絞り込み・検索機能 | **最優先** |
| F-04 | 手動データ更新 | ・**食材の消費**: 数量減算（ボタン操作）<br>・**食材の追加購入**: 数量加算<br>・**レシートがない場合の新規手動入力**: 食材名、数量、単位、賞味期限、保管場所を入力<br>・既存食材の編集・削除<br>・簡易操作UI（タップでの更新） | **優先** |
| F-06 | ユーザー認証 | ・ログイン機能（Google Auth）<br>・**認証**: NextAuth.js を使用<br>・機種変更時のデータ引き継ぎ（Webアプリのためログインのみで完了）<br>・家族間でのアカウント共有機能（1つの冷蔵庫を複数人で管理） | 必須 |
| F-10 | データ共有 | ・複数端末（家族間）でのデータ共有<br>・PostgreSQL を単一データソースとして使用（Webアプリのためリアルタイム同期は不要、都度API取得）<br>・競合解決ロジック（最終更新優先） | 必須 |

## 4. 技術選定とアーキテクチャ

### 4.1 決定済み技術

| カテゴリ | 採用技術 | 理由 |
|---|---|---|
| フレームワーク | **Next.js** (React / TypeScript) | SSR/SSG対応、APIルート内蔵、エコシステム最大、PWA対応可能 |
| データベース | **PostgreSQL** | RDBで食材管理との相性が良い、複合検索が容易、無料枠が充実したホスティング多数 |
| ORM | **Prisma** | スキーマファースト、型自動生成、マイグレーション自動生成、Next.jsとの統合実績豊富 |
| 認証 | **NextAuth.js (Auth.js)** | Next.jsネイティブ統合、Google Sign-In対応、ベンダーロックインなし、OSS |
| OCR | **Google Cloud Vision API** | 日本語レシートの認識精度が最高水準、1,000回/月無料、Webアプリからの利用が容易 |
| カメラ | **`<input type="file" capture="environment">`** | iOS Safari / Android Chrome 両対応、OS標準カメラを起動、実装コスト最小、最も安定 |
| 音声入力 | **Web Speech API** | ブラウザ内蔵・無料・日本語対応、iOS Safari（`webkitSpeechRecognition`）/ Android Chrome（`SpeechRecognition`）対応 |

### 4.2 インフラ・ライブラリ

| カテゴリ | 採用技術 | 理由 |
|---|---|---|
| ホスティング（アプリ） | **Vercel** | Next.jsの開発元、デプロイが最も簡単、プレビュー環境自動生成 |
| ホスティング（DB） | **Neon** | サーバーレスPostgreSQL、一時停止なし、ブランチング機能対応 |
| 画像ストレージ | **Cloudflare R2** | エグレス無料、10GB無料枠、S3互換API |
| 状態管理 | **Zustand + TanStack Query** | Zustand: UI状態管理、TanStack Query: サーバーデータのキャッシュ・再取得 |
| UIコンポーネント | **shadcn/ui** | コピペ型で完全カスタマイズ可能、Tailwind CSS統合 |
| CSSフレームワーク | **Tailwind CSS** | ユーティリティファースト、shadcn/uiと統合、モバイルファースト設計に最適 |

### 4.3 商品名正規化・カテゴリ判定ロジック

```
[レシート画像取得（カメラ / ギャラリー）]
    ↓
[Next.js API Route → Google Cloud Vision API]
    ↓
[正規化処理エンジン（サーバーサイド）]
    1. カタカナ統一変換
    2. 修飾語除去（辞書マッチング）
    3. 英語→日本語変換
    ↓
[カテゴリ判定エンジン]
    1. 商品マスタDBからカテゴリ情報取得
    2. キーワードマッチングでカテゴリを推定
    ↓
[保管場所自動設定]
    1. ユーザーのカテゴリ→保管場所マッピング参照
    2. 保管場所を自動設定
    ↓
[既存マスタDB検索]
    ↓
[ユーザー確認画面（保管場所変更可能）]
```

* **辞書データ**: PostgreSQLに以下のテーブルを用意
  * `normalization_rules`: 変換ルール（例: "ポーク" → "豚"）
  * `modifier_words`: 除去対象の修飾語（例: "おいしい", "こだわりの"）
  * `product_master`: 正規化済み商品名のマスタ（カテゴリ情報含む）
  * `category_storage_mapping`: ユーザーごとのカテゴリ→保管場所マッピング

## 6. データベース設計（主要テーブル）

### PostgreSQL（Prisma スキーマ）

```prisma
// schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// ユーザーテーブル（NextAuth.js管理）
model User {
  id            String    @id @default(cuid())
  name          String?
  email         String?   @unique
  emailVerified DateTime?
  image         String?
  accounts      Account[]
  sessions      Session[]
  foods         Food[]
  categoryStorageMappings CategoryStorageMapping[]
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

// NextAuth.js用テーブル
model Account {
  id                String  @id @default(cuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  refresh_token     String?
  access_token      String?
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String?
  session_state     String?
  user              User    @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
}

model Session {
  id           String   @id @default(cuid())
  sessionToken String   @unique
  userId       String
  expires      DateTime
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}

model VerificationToken {
  identifier String
  token      String   @unique
  expires    DateTime

  @@unique([identifier, token])
}

// 食材テーブル
model Food {
  id              String   @id @default(cuid())
  name            String   // 正規化済み食材名（カタカナ）
  originalName    String?  // 元の商品名（参考用）
  category        String?  // カテゴリ（野菜、肉、魚など）
  quantity        Float
  unit            String
  expiryDate      DateTime?
  storageLocation String?  // '冷蔵室','冷凍室','野菜室','常温'
  userId          String
  user            User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  @@index([userId])
  @@index([userId, storageLocation])
  @@index([userId, expiryDate])
}

// 正規化ルールテーブル
model NormalizationRule {
  id             String @id @default(cuid())
  originalWord   String
  normalizedWord String
  ruleType       String // 'kana_conversion','english_to_japanese','modifier_removal'

  @@index([originalWord])
}

// 商品マスタテーブル
model ProductMaster {
  id             String  @id @default(cuid())
  normalizedName String  @unique
  category       String
  defaultUnit    String?
  avgExpiryDays  Int?
}

// カテゴリ→保管場所マッピングテーブル（ユーザー設定）
model CategoryStorageMapping {
  id              String @id @default(cuid())
  userId          String
  category        String
  storageLocation String // '冷蔵室','冷凍室','野菜室','常温'
  user            User   @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, category])
}

// カテゴリマスタテーブル
model Category {
  id        String @id @default(cuid())
  name      String @unique
  icon      String?
  sortOrder Int?
}
```

## 7. 開発方針

### プロトタイプ駆動開発

本プロジェクトは以下の方針で進めます:

1. **Phase 1でプロトタイプを完成**
   - レシートOCR + 食材一覧表示 + 手動更新 + 認証 + データ共有の機能を実装
   - 実際に動作する状態を早期に作成
   - ユーザーフィードバックを収集

2. **段階的な機能追加**
   - プロトタイプをベースに、Phase 2、Phase 3の機能を追加
   - 各フェーズで実際に使いながら改善
   - 優先度の高い機能から実装

3. **継続的な改善**
   - 実使用での課題を発見・修正
   - OCR精度、正規化ルールの継続的な改善
   - ユーザー体験の最適化

### 最小プロトタイプの定義

以下の機能が動作すればプロトタイプとして完成とみなします:

- レシートを撮影（またはギャラリーから選択）してOCRで商品を読み取る
- 商品名を正規化してDBに保存
- 食材一覧を表示（賞味期限順、保管場所別）
- 手動で食材の消費・追加・編集が可能
- Google認証でログイン
- 複数端末でデータ共有（同一アカウントでアクセス）

### プロジェクト管理

* **バージョン管理**: GitHub
  * リポジトリ構成: Next.js フルスタックアプリとして同一リポジトリで管理
  * ブランチ戦略: Git Flow を採用
    * `main`: 本番環境
    * `develop`: 開発環境
    * `feature/*`: 機能開発
    * `release/*`: リリース準備
    * `hotfix/*`: 緊急修正

* **プロジェクト管理**: Atlassian Jira
  * Epic: Phase 1, Phase 2, Phase 3
  * Story: 各機能（F-01, F-02など）
  * Task: 実装タスク、テストタスク
  * スプリント期間: 2週間
  * 進捗管理: Kanbanボードとバーンダウンチャート

## 8. 変更履歴

### ✅ v3.0での変更内容（ネイティブアプリ → Webアプリへのシフト）

1. **提供形態の変更**
   - Flutter（ネイティブアプリ）→ Next.js（Webアプリ / PWA）
   - 理由: iOSでの無料配布が困難（Apple Developer Program年額$99）。Webアプリなら無料で継続運用可能
   - iOS: Safari、Android: Chrome を主対応ブラウザとし、PC は Chrome（今後 Safari / Edge にも対応予定）

2. **技術スタック全面変更**
   - フレームワーク: Flutter → **Next.js (TypeScript)**
   - データベース: Firestore + SQLite → **PostgreSQL**（単一DB構成に簡素化）
   - ORM: なし → **Prisma**（型安全なDB操作）
   - 認証: Firebase Auth → **NextAuth.js**
   - OCR: Google ML Kit → **Google Cloud Vision API**（サーバーサイド処理）

3. **アーキテクチャの簡素化**
   - ローカルDB + クラウドDB の二重構成 → PostgreSQL単一構成
   - オフライン対応はPhase 3で検討（Service Worker + IndexedDB）
   - リアルタイム同期は不要（Webアプリのため都度API取得で十分）

4. **カメラ・音声の実現方式変更**
   - カメラ: ネイティブカメラAPI → `<input type="file" capture="environment">` によるOS標準カメラ起動
   - 音声: Flutter Speech-to-Text → Web Speech API（`webkitSpeechRecognition` / `SpeechRecognition`）

5. **データベース設計の変更**
   - Prisma スキーマとして再定義
   - NextAuth.js用テーブル（User, Account, Session, VerificationToken）を追加
   - Firestoreのドキュメント構成を廃止し、PostgreSQLのリレーショナル設計に統一

### ✅ v2.2での変更内容

1. **保管場所管理（F-05）の大幅拡張**
   - Phase 1から Phase 3の最初に移動
   - 初期設定画面の追加（カテゴリごとの保管場所設定）
   - OCRデータの自動カテゴライズ機能
   - 手動変更機能の追加
   - データベースに `category_storage_mapping` テーブルを追加

2. **データ同期（F-10）の優先度変更**
   - Phase 3から Phase 1（コア機能）の最後に移動
   - 家族間でのデータ共有を早期実現

3. **検索・絞り込み機能（F-13）の追加**
   - Phase 3に新機能として追加
   - カテゴリでの絞り込み
   - キーワードでの部分一致検索
   - 複合検索機能

4. **プロジェクト管理ツールの明記**
   - GitHub: バージョン管理、Git Flow戦略
