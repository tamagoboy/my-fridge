# 冷蔵庫管理アプリ 要件定義書 (Draft 4.0)

## 1. プロジェクト概要

* **目的**: 冷蔵庫内の食材を管理し、賞味期限切れによる廃棄を防ぐとともに、買い物の効率化を図る。
* **ターゲット**: 自炊を行う単身者およびファミリー層。
* **コア価値**: 「入力の手間を極限まで減らす（スマート入力）」こと。

## 2. システム構成・対応プラットフォーム

* **提供形態**: Webアプリ（PWA対応）
  * **採用技術**: Next.js (React / TypeScript)
  * **変更理由**: iOSでは個人開発者がアプリを無料で配布し続けることが困難（Apple Developer Program年額$99 + 審査コスト）。Webアプリ（PWA）にすることで、ストア審査不要・無料で継続運用が可能。
  * **PWAによるネイティブ相当の体験**: ホーム画面追加、オフライン対応、プッシュ通知（iOS Safari 16.4+ / Android Chrome 対応）
* **対応ブラウザ**:
  * **iOS**: Safari（PWA機能をフル活用するため。iOSでは全ブラウザがWebKitエンジンのため、Safari が最も機能制限が少ない）
  * **Android**: Google Chrome
  * **PC**: Google Chrome（今後 Safari / Microsoft Edge にも対応予定）
  * **方針**: クロスブラウザ互換性を意識した実装を行い、段階的にブラウザ対応を拡大
* **レスポンシブ対応**: モバイルファースト設計

## 3. 機能要件 (フェーズ別)

### Phase 1: コア機能（プロトタイプ / MVP）

このフェーズではレシートOCRを中心とした「実用的な最小機能」を完成させます。プロトタイプとして動作する状態を作り、そこから段階的に機能を追加していきます。

| 機能ID | 機能名 | 詳細要件 | 優先度 |
|---|---|---|---|
| F-01 | レシートOCR入力 | ・カメラまたはギャラリーからレシート画像を取得<br>・OCR処理で商品名を自動認識（数量・単位はレシートから読み取らない）<br>・**OCRエンジン**: Google Cloud Vision API を使用（サーバーサイド処理）<br>・**カメラ**: `<input type="file" capture="environment">` でOS標準カメラを起動<br>・認識後、即座に商品名正規化処理を実行<br>・**AI による保管場所自動分類**: OCR 結果を AI に渡し、各食材を冷蔵庫/冷凍庫/常温に自動分類<br>・残量は全て「いっぱい」で初期登録<br>・確認モーダルで編集可能（食材名・保管場所・残量の変更、不要アイテムの削除）<br>・DBへの一括保存 | **最優先** |
| F-02 | 商品名正規化処理 | **OCR読み取り直後に以下の処理をサーバーサイドで自動実行**:<br>1. ひらがな・カタカナを全てカタカナに統一<br>2. 店舗固有の修飾語を除去（例: 「おいしい豚肉」→「豚肉」）<br>3. 英語表記を日本語に変換（例: 「ポーク」→「豚」、「ビーフ」→「牛」）<br>・変換辞書はPostgreSQLに保持<br>・正規化後の商品名でDB検索し、既存マスタとマッチング<br>・**LLMフォールバック**: ルールベース辞書でマッチしない場合のみ LLM に正規化を委譲（例: 「黒豚モモスライス300g」→「豚肉」） | **最優先** |
| F-03 | 食材一覧表示 | ・保管場所別タブ（デフォルト: 冷蔵庫/冷凍庫/常温、ユーザーが自由に追加可能）<br>・登録日順での表示<br>・残量ステータス（いっぱい/半分/少し）をアイコンで表示<br>・基本的な絞り込み・検索機能 | **最優先** |
| F-04 | 手動データ更新 | ・**食材の消費**: 残量ステータスの変更（いっぱい→半分→少し、またはスワイプで消費済み削除）<br>・**レシートがない場合の新規手動入力**: 食材名、保管場所を入力（残量はデフォルト「いっぱい」）<br>・既存食材の編集・削除<br>・簡易操作UI（タップでの更新） | **優先** |
| F-06 | ユーザー認証 | ・ログイン機能（Google Auth のみ）<br>・**認証**: NextAuth.js を使用（**JWT 戦略**、Session テーブル不要）<br>・`googleId`（Google の sub）を User に直接保持（PrismaAdapter・Account テーブル不要）<br>・初回ログイン時に冷蔵庫（Fridge）を自動作成<br>・機種変更時のデータ引き継ぎ（Webアプリのためログインのみで完了） | 必須 |
| F-10 | データ共有 | ・複数ユーザーが **1つの Fridge グループ** を共有<br>・食材は User ではなく **Fridge に紐づく**（`Food.fridgeId`）<br>・保管場所も **Fridge に紐づく**（`StorageLocation.fridgeId`）<br>・**招待リンク**（UUID トークン、7日間有効）で家族・ルームメイトを招待<br>・1ユーザー1冷蔵庫（複数冷蔵庫への同時所属不可）<br>・全メンバー同等権限（オーナー/メンバーの区別なし）<br>・PostgreSQL を単一データソースとして使用（リアルタイム同期不要）<br>・消費操作は `updatedAt` による楽観的ロックで競合解決<br>・招待 API・UI は Phase 1 後半で実装 | 必須 |
| F-14 | AIレシピベース消費 | ・ユーザーが「何を作るか（レシピ名）」を入力<br>・AI がユーザーの冷蔵庫の食材データベースから、そのレシピに使いそうな食材を推定<br>・確認モーダルで推定された食材一覧を表示<br>・ユーザーが食材の追加・削除・残量変更を編集可能<br>・確定後に一括で残量ステータスを更新（少し→削除、半分→少し 等）<br>・**AI モデル**: 未選定（Gemini が候補、無料枠が大きいため） | 優先 |

### Phase 2: スマート入力・期限管理

入力の効率化と賞味期限管理を実装します。

| 機能ID | 機能名 | 詳細要件 |
|---|---|---|
| F-05 | 賞味期限管理 | **手動入力**:<br>・手動入力画面・編集画面で賞味期限（日付）を任意入力可能<br>・レシートOCRからの自動読み取りは困難なため対応しない<br><br>**期限切迫度カラー表示**:<br>・食材一覧で期限切迫度に応じたカラー表示（赤/黄/緑）<br>・赤: 期限切れ・当日、黄: 3日以内、緑: 4日以上<br><br>**プッシュ通知との連携**:<br>・賞味期限3日前・当日に通知（F-11と連動） |
| F-15 | 賞味期限自動推定 | ・食材登録時に食材名を LLM に送信し、一般的な賞味期限を推定して初期値として提案<br>・例: 「牛乳」→ 開封後3〜4日、「豚肉（生）」→ 冷蔵1〜2日 / 冷凍2〜3週間<br>・ユーザーは提案値を確認・変更可能（任意入力）<br>・F-05（賞味期限管理）の入力補助として機能<br>・**AI モデル**: F-01 と同じモデルを使用 |
| F-08 | 入力履歴サジェスト | ・過去に入力した食材を履歴として保持<br>・入力時にサジェスト（入力予測）を表示<br>・使用頻度順に並び替え<br>・学習機能で精度向上 |

## 4. 技術選定とアーキテクチャ

### 4.1 決定済み技術

| カテゴリ | 採用技術 | 理由 |
|---|---|---|
| フレームワーク | **Next.js** (React / TypeScript) | SSR/SSG対応、APIルート内蔵、エコシステム最大、PWA対応可能 |
| データベース | **PostgreSQL** | RDBで食材管理との相性が良い、複合検索が容易、無料枠が充実したホスティング多数 |
| ORM | **Prisma** | スキーマファースト、型自動生成、マイグレーション自動生成、Next.jsとの統合実績豊富 |
| 認証 | **NextAuth.js (Auth.js)** | Next.jsネイティブ統合、Google Sign-In対応、ベンダーロックインなし、OSS |
| OCR | **Google Cloud Vision API** | 日本語レシートの認識精度が最高水準、1,000回/月無料、Webアプリからの利用が容易 |
| AI（食材分類・消費推定） | **未選定**（Gemini が第一候補） | OCR 後の保管場所自動分類、レシピベース消費推定に使用。Gemini は無料枠が大きい。他に適切なモデルがあれば変更可能 |
| カメラ | **`<input type="file" capture="environment">`** | iOS Safari / Android Chrome 両対応、OS標準カメラを起動、実装コスト最小、最も安定 |
| 音声入力 | **Web Speech API** | ブラウザ内蔵・無料・日本語対応、iOS Safari（`webkitSpeechRecognition`）/ Android Chrome（`SpeechRecognition`）対応 |

### 4.4 レシートOCR + AI分類フロー

```
[レシート画像取得（カメラ / ギャラリー）]
    ↓
[Next.js API Route → Google Cloud Vision API]
    ↓
[正規化処理エンジン（サーバーサイド）]
    1. カタカナ統一変換
    2. 修飾語除去（コード内辞書）
    3. 英語→日本語変換
    ↓
[AI 保管場所自動分類]
    1. 正規化済み食材名リストを AI に送信
    2. 各食材を冷蔵庫/冷凍庫/常温（+ ユーザーのカスタム保管場所）に分類
    3. 残量は全て「いっぱい」で初期設定
    ↓
[確認モーダル（編集可能）]
    ・食材名の編集
    ・保管場所の変更
    ・残量の変更
    ・不要アイテムの削除
    ↓
[一括DB保存]
```

### 4.5 AIレシピベース消費フロー（Phase 1）

```
[ユーザーがレシピ名を入力（例: 「カレー」）]
    ↓
[AI にレシピ名 + ユーザーの冷蔵庫内食材リストを送信]
    ↓
[AI が使いそうな食材を推定・返却]
    ↓
[確認モーダル（編集可能）]
    ・推定された食材の確認
    ・食材の追加・削除
    ・残量変更の指定（半分→少し、少し→削除 等）
    ↓
[確定 → 一括で残量ステータス更新]
```

## 6. データベース設計（主要テーブル）

### PostgreSQL（Prisma スキーマ）

> 詳細設計: `docs/steps/step1-db-schema.md`

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// ─────────────────────────────────────────────
// 認証テーブル（Google OAuth のみ対応）
// ─────────────────────────────────────────────

// - セッション管理は JWT 戦略（Cookie）→ Session テーブル不要
// - Google のみ対応 → Account テーブル不要、googleId を User に直接持つ
// - メール認証なし → VerificationToken テーブル不要
model User {
  id       String  @id @default(cuid())
  googleId String  @unique // Google の sub フィールド
  name     String?
  email    String  @unique
  image    String? // Google プロフィール画像 URL

  fridgeMember FridgeMember?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

// ─────────────────────────────────────────────
// 冷蔵庫共有テーブル
// ─────────────────────────────────────────────

// 冷蔵庫グループ（初回ログイン時に自動作成）
model Fridge {
  id   String @id @default(cuid())
  name String @default("マイ冷蔵庫")

  members          FridgeMember[]
  foods            Food[]
  invitations      FridgeInvitation[]
  storageLocations StorageLocation[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

// 冷蔵庫メンバー（1 ユーザー 1 冷蔵庫を保証）
model FridgeMember {
  id       String @id @default(cuid())
  userId   String @unique
  fridgeId String

  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)
  fridge Fridge @relation(fields: [fridgeId], references: [id], onDelete: Cascade)

  joinedAt DateTime @default(now())

  @@index([fridgeId])
}

// 招待リンク管理（UUID トークン、7 日間有効、冷蔵庫につき 1 リンク）
model FridgeInvitation {
  id        String   @id @default(cuid())
  fridgeId  String
  token     String   @unique @default(uuid())
  expiresAt DateTime

  fridge Fridge @relation(fields: [fridgeId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now())
}

// ─────────────────────────────────────────────
// アプリケーションテーブル
// ─────────────────────────────────────────────

// 保管場所（Fridge に紐づく、ユーザーが自由に追加可能）
// 初回ログイン時にデフォルト 3 つ（冷蔵庫/冷凍庫/常温）を自動作成
model StorageLocation {
  id        String @id @default(cuid())
  name      String   // 例: "冷蔵庫", "冷凍庫", "常温", "野菜室", "パントリー"
  sortOrder Int    @default(0)
  fridgeId  String

  fridge Fridge @relation(fields: [fridgeId], references: [id], onDelete: Cascade)
  foods  Food[]

  @@unique([fridgeId, name])
  @@index([fridgeId])
}

// 食材テーブル（食材は Fridge に紐づく）
model Food {
  id                String  @id @default(cuid())
  name              String  // 食材名
  category          String? // カテゴリ名（野菜/肉/魚/惣菜/乳製品/調味料/その他）
  remainingStatus   String  @default("full") // 残量: 'full'|'half'|'little'
  storageLocationId String? // StorageLocation への外部キー
  fridgeId          String

  fridge          Fridge           @relation(fields: [fridgeId], references: [id], onDelete: Cascade)
  storageLocation StorageLocation? @relation(fields: [storageLocationId], references: [id], onDelete: SetNull)

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([fridgeId])
  @@index([fridgeId, storageLocationId])
}

// ─────────────────────────────────────────────
// Phase 3 以降で追加予定
// ─────────────────────────────────────────────
// - NormalizationRule: 正規化はバックエンドコードで実装（Step 6）
// - ProductMaster: 同上
// - expiryDate を Food に追加: Phase 3（F-05 賞味期限管理）で実装
```

## 7. 開発方針

### プロトタイプ駆動開発

本プロジェクトは以下の方針で進めます:

1. **Phase 1でプロトタイプを完成**
   - レシートOCR + 食材一覧表示 + 手動更新 + 認証 + データ共有の機能を実装
   - 実際に動作する状態を早期に作成
   - ユーザーフィードバックを収集

### 最小プロトタイプの定義

以下の機能が動作すればプロトタイプとして完成とみなします:

- レシートを撮影（またはギャラリーから選択）してOCRで商品を読み取る
- 商品名を正規化し、AIが保管場所を自動分類
- 確認モーダルで編集後、DBに一括保存
- 食材一覧を表示（保管場所別タブ、残量ステータス表示）
- 手動で食材の残量変更・追加・編集・削除が可能
- Google認証でログイン
- 複数端末でデータ共有（同一アカウントでアクセス）
- カスタム保管場所の追加が可能
- レシピ名を入力してAIが使用食材を推定し、確認後に一括消費できる

### ✅ v3.0での変更内容（ネイティブアプリ → Webアプリへのシフト）

1. **提供形態の変更**
   - Flutter（ネイティブアプリ）→ Next.js（Webアプリ / PWA）
   - 理由: iOSでの無料配布が困難（Apple Developer Program年額$99）。Webアプリなら無料で継続運用可能
   - iOS: Safari、Android: Chrome を主対応ブラウザとし、PC は Chrome（今後 Safari / Edge にも対応予定）

2. **技術スタック全面変更**
   - フレームワーク: Flutter → **Next.js (TypeScript)**
   - データベース: Firestore + SQLite → **PostgreSQL**（単一DB構成に簡素化）
   - ORM: なし → **Prisma**（型安全なDB操作）
   - 認証: Firebase Auth → **NextAuth.js**
   - OCR: Google ML Kit → **Google Cloud Vision API**（サーバーサイド処理）

3. **アーキテクチャの簡素化**
   - ローカルDB + クラウドDB の二重構成 → PostgreSQL単一構成
   - オフライン対応はPhase 3で検討（Service Worker + IndexedDB）
   - リアルタイム同期は不要（Webアプリのため都度API取得で十分）

4. **カメラ・音声の実現方式変更**
   - カメラ: ネイティブカメラAPI → `<input type="file" capture="environment">` によるOS標準カメラ起動
   - 音声: Flutter Speech-to-Text → Web Speech API（`webkitSpeechRecognition` / `SpeechRecognition`）

5. **データベース設計の変更**
   - Prisma スキーマとして再定義
   - NextAuth.js用テーブル（User, Account, Session, VerificationToken）を追加
   - Firestoreのドキュメント構成を廃止し、PostgreSQLのリレーショナル設計に統一
