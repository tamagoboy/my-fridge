# 冷蔵庫管理アプリ 実装手順書 v2.0

> 要件定義書 v3.0（Next.js / PostgreSQL / Prisma）に準拠

## 基本方針

- **安定した土台を先に作る**: DB・認証・UI の基盤を固めてから OCR 等の複雑な機能を追加
- **動作確認しながら積み上げる**: 各ステップで実際に動く状態を保つ
- **依存関係の少ない順に実装**: 後から機能を追加しやすい構造を優先

## Step 6: 商品名正規化処理

**対応機能**: F-02

### 6-1. 正規化エンジン

**`src/lib/normalizer.ts`**

OCR 読み取り直後にサーバーサイドで自動実行:

1. **カタカナ統一**: ひらがな → カタカナに変換
2. **修飾語除去**: コード内の修飾語リストにマッチする語を除去（DB 辞書は使わない）
3. **英語→日本語変換**: コード内の変換マップで変換（DB 辞書は使わない）
4. **カテゴリ推定**: 食材名のキーワードマッチングでカテゴリを推定（コードベース）

### 6-2. 正規化 API

**`src/app/api/ocr/route.ts`** 内に組み込み

- OCR 結果 → 正規化 → カテゴリ判定 → 保管場所自動設定
- 一連のパイプラインとして処理

### 6-3. OCR フローへの組み込み

```
レシート画像 → Cloud Vision API → テキスト抽出
  → 正規化エンジン → カテゴリ判定 → 保管場所自動設定
  → 確認画面に表示
```

**確認**: 「おいしいポーク」→「豚」（カテゴリ: 肉）に変換されること

## Step 7: データ共有（冷蔵庫共有・招待機能）

**対応機能**: F-10

> 詳細設計: `docs/steps/step1-fridge-sharing.md`

### 7-1. 設計方針（DB 基盤は Step 1 で完了）

- 食材は `Fridge` に紐づく（`Food.fridgeId`）
- PostgreSQL を単一データソースとして使用（リアルタイム同期不要）
- 同一 Fridge のメンバーなら、どの端末・アカウントからでもデータを共有

### 7-2. 招待リンク機能（Step 7 の主実装）

- **`POST /api/fridge/invite`**: 招待リンク発行（UUID トークン、7 日間有効）
- **`GET /api/fridge/invite/[token]`**: 招待情報取得（確認画面用）
- **`POST /api/fridge/join/[token]`**: 冷蔵庫に参加（既存冷蔵庫から離脱→新冷蔵庫に参加）
- **`POST /api/fridge/leave`**: 冷蔵庫から離脱（新しい空の冷蔵庫を自動作成）
- **`GET /api/fridge/members`**: メンバー一覧取得

### 7-3. 招待 UI

- 設定画面にメンバー一覧・「招待リンクをコピー」ボタン・「離脱」ボタンを追加
- `/invite/[token]` ページ: 冷蔵庫名と「参加する」ボタンを表示

### 7-4. 競合解決

- 消費操作は `updatedAt` による楽観的ロック
  - `WHERE id = ? AND updatedAt = ?` で競合を検知
  - 競合時はエラーを返してクライアントに再取得を促す

**確認**: 招待リンクで別アカウントが同じ冷蔵庫に参加し、食材データが共有されること
